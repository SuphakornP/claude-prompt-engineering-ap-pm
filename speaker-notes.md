# Speaker Notes

## Session Frame

ผู้ฟัง: Project Manager ของ AP Thailand

เป้าหมาย: ให้ PM ใช้ Claude เป็นผู้ช่วยทำงานจริง โดยเขียน prompt แบบ brief ที่ชัด เลือกประเภทการใช้งานให้ตรงงาน และรู้ว่าเมื่อไรควรใช้ Chat, Cowork หรือ Code

แกนการเล่า:

- Prompt ที่ดีไม่ใช่ประโยคยาว แต่คือ brief ที่คนอ่านแล้วทำงานต่อได้
- งาน PM คือการรวมข้อมูลหลายแหล่งให้กลายเป็นการตัดสินใจ
- Claude มี 3 ประเภทการใช้งานหลัก: Chat, Cowork, Code
- เลือกโมเดลตามความยาก ความเร็ว และความเสี่ยงของงาน
- ทุก prompt ควรมีงาน บริบท แหล่งข้อมูล รูปแบบคำตอบ และเกณฑ์ตรวจ

หมายเหตุชื่อทางการ:

- ใช้ `Claude Cowork` ไม่ใช้ `Claude Coworker`
- ใช้ `Haiku` ไม่ใช้ `Hiku`
- `Mythos 5` มีใน official docs แต่ยังเป็น limited availability ผ่าน Project Glasswing จึงไม่ควรสอนเป็นตัวเลือกหลักของทีม

## 1. เปิดเรื่อง

เปิดด้วยประโยคง่าย ๆ:

"วันนี้เราไม่ได้มาเรียนสูตร prompt สำเร็จรูป แต่จะเรียนวิธี brief งานให้ Claude เหมือน brief เพื่อนร่วมทีมคนหนึ่ง"

อธิบายว่า PM ไม่ได้อยากได้คำตอบสวย ๆ อย่างเดียว แต่ต้องการผลลัพธ์ที่เอาไปใช้ต่อได้ เช่น สรุปประชุม ตารางเทียบ vendor memo สำหรับตัดสินใจ หรือ prototype ให้ทีมเห็นภาพ

เชื่อมไปหน้าถัดไป: "ก่อนเข้าเนื้อหา เราดูภาพรวมกันก่อนว่าวันนี้จะผ่านอะไรบ้าง"

## 2. Agenda

ใช้หน้านี้บอกภาพรวมแบบเร็ว ๆ ไม่ต้องอธิบายทุก bullet ยาว เพราะแต่ละหัวข้อจะมีสไลด์ของตัวเอง

Talk track:

"วันนี้มี 3 เรื่องหลัก: เลือกประเภทของ Claude ให้ตรงงาน เลือก Model ให้พอดี และเขียน Prompt ที่ดีให้ Claude ทำงานได้ชัด"

"ถ้าจำได้แค่อย่างเดียว ให้จำว่า Claude จะเก่งขึ้นเมื่อ PM brief งานชัดขึ้น ไม่ใช่แค่พิมพ์คำสั่งยาวขึ้น"

เชื่อมไปหน้าถัดไป: "เริ่มจาก mindset แรกก่อน คือ prompt ไม่ใช่คาถา แต่คือ brief"

## 3. Prompt คือ Brief

เล่า quote จากคุณ Piyalitt ว่า ถ้าเอา prompt ให้คนที่ไม่รู้บริบทอ่านแล้วเขายังงง Claude ก็จะงงเหมือนกัน

ชวนผู้ฟังคิดถึงการ brief vendor หรือทีม internal ถ้าไม่บอกเป้าหมาย ข้อมูลอ้างอิง deadline หรือรูปแบบงาน คนทำงานก็ต้องเดา AI ก็เดาเช่นกัน

ประโยคสำคัญ: "Prompt ที่ดีวัดจากว่าอีกฝ่ายทำตามได้ไหม ไม่ได้วัดจากความยาว"

## 4. งานจริงของ PM

ย้ำว่า PM ต้องเชื่อม minutes, issue log, timeline, RFI, vendor reply, budget และ owner expectation พร้อมกัน

ตัวอย่าง: issue หนึ่งข้ออาจกระทบ schedule, budget, quality และการตัดสินใจของผู้บริหารพร้อมกัน Claude ช่วยได้เมื่อเราให้ข้อมูลต้นทางและสั่งให้จัดรูปเป็นข้อเสนอสำหรับ PM ตรวจ

เชื่อมไปหน้าถัดไป: "เมื่อรู้ว่างานมีหลายแบบ เราต้องเลือกประเภทการใช้งาน Claude ให้ถูกก่อน"

## 5. ประเภทการใช้งาน Claude

อธิบายแบบจำง่าย:

- Chat คือเพื่อนคู่คิด ใช้คุย ถามต่อ สรุปไฟล์ และแก้คำไปทีละรอบ
- Cowork คือผู้ช่วยจัดงาน ใช้เมื่อมีหลายไฟล์ หลายขั้นตอน หรืออยากให้ Claude ทำงานจนได้ชิ้นงานให้ตรวจ
- Code คือตัวทำของทดลองใช้ ใช้เมื่ออยากได้ tracker, dashboard, prototype หรือไฟล์ที่ลองเปิดได้

ประโยคสำคัญ: "อย่าเริ่มจากถามว่า prompt ยังไง ให้เริ่มจากถามว่างานนี้ควรออกมาเป็นอะไร"

## 6. เลือกโมเดล

อธิบายให้เป็นภาษาคนทำงาน:

Sonnet 5 คือจุดเริ่มต้นที่ดีสำหรับงาน PM ส่วนใหญ่ เพราะสมดุลระหว่างความฉลาดและความเร็ว

Haiku 4.5 เหมาะกับงานเร็ว งานง่าย และจำนวนมาก เช่น extract ข้อมูลหรือจัดหมวดหมู่ issue

Opus 4.8 เหมาะกับงานวิเคราะห์ลึก อ่านหลายแหล่ง หรือมีความเสี่ยงสูง

Fable 5 เหมาะกับงานซับซ้อนมาก งาน agent ยาว หรืองานที่ต้องการความสามารถสูงสุดและยอมรับต้นทุนได้

ย้ำว่า Mythos 5 มีใน official docs แต่ยังไม่ใช่ตัวเลือกที่ควรสอนให้ทีมทั่วไปใช้เป็นค่าเริ่มต้น

## 7. โครงสร้าง Prompt

บอกทีมว่า prompt ที่ดีควรมี 6 ส่วน:

- บทบาท: ให้ Claude คิดจากมุมไหน
- เป้าหมาย: งานที่ต้องทำคืออะไร
- บริบท: ฉากหลังของ project คืออะไร
- ข้อมูลอ้างอิง: ให้ใช้ไฟล์หรือข้อมูลชุดไหน
- รูปแบบคำตอบ: อยากได้ memo ตาราง checklist หรือ format อื่น
- เกณฑ์ตรวจ: อะไรคือคำตอบที่ดีพอ

ชี้ตัวอย่างบนสไลด์ แล้วถามผู้ฟังว่า ถ้าตัด "ข้อมูลอ้างอิง" หรือ "เกณฑ์ตรวจ" ออก Claude จะเริ่มเดาตรงไหนบ้าง

## 8. เอกสารยาว

บอกหลักง่าย ๆ: ถ้างานมีเอกสารยาว ให้ใส่เอกสารต้นทางก่อน แล้วค่อยสั่งงานท้าย prompt

ตัวอย่าง PM: วาง minutes, vendor reply และ timeline แยกเป็นแหล่งข้อมูลชัด ๆ แล้วค่อยถามว่า "สรุปผลกระทบ ความเสี่ยง เจ้าของงาน และเรื่องที่ต้องตัดสินใจในการประชุมถัดไป"

ย้ำให้ตั้งชื่อแหล่งข้อมูล เช่น `minutes`, `vendor reply`, `timeline` เพื่อให้ Claude อ้างอิงกลับได้

## 9. ให้ตัวอย่าง

อธิบายว่า ถ้าอยากคุมคุณภาพ ให้ Claude เห็นตัวอย่างงานที่ดี ไม่ใช่บรรยายความต้องการอย่างเดียว

ตัวอย่างบนสไลด์:

- สั่งกว้าง: "ช่วยเขียนสรุปสถานะให้ผู้บริหารจากรายการประเด็นนี้"
- สั่งดีกว่า: ให้ Claude ดูตัวอย่างงานที่ดี 1 ชิ้นก่อน เช่น สถานะ ประเด็น ผลกระทบ เรื่องที่ต้องตัดสินใจ และเจ้าของงาน จากนั้นใส่ประเด็นใหม่แล้วบอกให้ใช้รูปแบบเดียวกัน

แนะนำให้มีตัวอย่าง 3 แบบ: สถานะปกติ, เริ่มเสี่ยง, ต้อง escalate เพื่อให้ Claude เห็น pattern ที่ครบกว่า

## 10. สั่งให้ทำชิ้นงาน

อธิบายความต่างระหว่าง "ขอความเห็น" กับ "ให้ทำงาน"

ถ้าถามว่า "ช่วยแนะนำวิธีจัด issue log" Claude จะตอบเป็นแนวทาง ซึ่งไม่ผิด แต่ PM ยังต้องทำต่อเอง

ถ้าต้องการชิ้นงาน ให้สั่งชัด เช่น "อ่าน issue log นี้ แล้วทำตาราง 3 คอลัมน์: ประเด็น / เจ้าของ / สิ่งที่ต้องตัดสินใจ"

ย้ำว่าเมื่องานเกี่ยวกับไฟล์จริง ให้กำหนด folder หรือข้อมูลนำเข้าให้ชัด และให้ PM ตรวจชิ้นงานก่อนส่งต่อ

## 11. เลือกใช้ให้ตรงงาน

ใช้ตารางนี้เป็น cheat sheet:

- สรุปประชุมหรือ RFI: เริ่มที่ Claude Chat + Projects
- เทียบ vendor proposal หลายไฟล์: ใช้ Claude Cowork
- รายงานซ้ำประจำสัปดาห์: ใช้ Cowork แบบตั้งงานซ้ำ
- Tracker, dashboard, prototype: ใช้ Claude Code
- งานสำคัญและเสี่ยงสูง: ใช้โมเดลที่แรงขึ้น และให้ PM ตรวจซ้ำ

ประโยคสำคัญ: "งานคนละแบบ ไม่จำเป็นต้องใช้ Claude ประเภทเดียวกัน"

## 12. Claude Chat: เพื่อนคู่คิด

อธิบายตาม official framing ว่า Chat ใช้เขียน แก้ วิเคราะห์ไฟล์หรือรูป ค้นเว็บ สร้างไฟล์ และทำ Artifacts ได้

เหมาะกับงานที่ PM ยังอยากคุมทิศทางเอง เช่น ถามต่อ ขอ version ใหม่ ปรับ tone หรือให้ช่วยคิดในมุม stakeholder

แนะนำ Projects สำหรับงานที่มีบริบทเดิมซ้ำ ๆ เช่น project เดียวกัน เพราะไม่ต้องเริ่มอธิบายใหม่ทุกครั้ง

## 13. Claude Cowork: ผู้ช่วยจัดงาน

อธิบายว่า Cowork เหมาะกับงานที่ยาวกว่า chat ปกติ เช่น มีหลายไฟล์ หลายขั้นตอน หรืออยากให้ Claude ไปทำงานแล้วกลับมาพร้อมชิ้นงานให้ตรวจ

ตัวอย่างที่ทีม PM เห็นภาพ:

- รวม minutes, issue log และ schedule เป็นรายงานประจำสัปดาห์
- อ่าน vendor proposal หลายไฟล์แล้วทำตารางเปรียบเทียบ
- รวมไฟล์ในโฟลเดอร์เพื่อเตรียมชุดข้อมูลสำหรับประชุมผู้บริหาร

ย้ำให้เลือกไฟล์หรือ folder ให้ถูก และ review ก่อนนำผลลัพธ์ไปส่งต่อ

## 14. Claude Code: ทำของทดลองใช้

ลดความกังวลว่า PM ไม่จำเป็นต้องเป็น developer เพื่อใช้ Code ในงานเบื้องต้น

ใช้ Code เมื่ออยากให้ prompt กลายเป็นไฟล์หรือเครื่องมือที่ลองเปิดได้ เช่น issue tracker จาก CSV, prototype จาก PRD, checklist QA หรือ dashboard ง่าย ๆ

ย้ำว่า Code เหมาะกับของทดลองใช้และการคุยกับทีมเทคนิคให้เห็นภาพ ไม่ใช่การ bypass engineering review สำหรับ production

## 15. สรุป

ปิดด้วย 3 ข้อ:

1. บอกงานให้ชัดว่าอยากให้ Claude ทำอะไร
2. ให้บริบทและข้อมูลอ้างอิงที่เกี่ยวข้อง
3. กำหนดผลลัพธ์ที่ PM ใช้ต่อได้ เช่น ตาราง memo checklist หรือไฟล์ทดลองใช้

Closing line:

"Prompt ที่ดีคือ brief ที่ชัด และ PM ที่ใช้ Claude ได้ดีคือ PM ที่ให้บริบทดี ตรวจผลลัพธ์เป็น และเลือกเครื่องมือให้ตรงกับงาน"
